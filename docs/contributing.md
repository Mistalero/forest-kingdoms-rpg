# Руководство по вкладу в проект

## Добро пожаловать!

Спасибо за ваш интерес к проекту Forest Kingdoms RPG! Мы рады любому вкладу от сообщества, будь то исправление ошибок, улучшение документации или добавление новых функций.

## Кодекс поведения

Прежде чем начать работу над проектом, ознакомьтесь с нашим [Кодексом поведения](CODE_OF_CONDUCT.md). Мы ожидаем, что все участники будут уважать его положения.

## Как внести вклад

### Сообщения об ошибках

#### Перед тем как создать issue
1. Проверьте [существующие issues](https://sourcecraft.dev/p2p-decentralized-emulation/forest-kingdoms-rpg/issues), чтобы убедиться, что ошибка еще не сообщена
2. Попробуйте воспроизвести ошибку в последней версии проекта
3. Убедитесь, что ошибка действительно связана с кодом проекта, а не с вашей локальной настройкой

#### Создание issue
При создании нового issue о ошибке, включите следующую информацию:
- **Краткое описание** - Четкое и краткое описание проблемы
- **Шаги для воспроизведения** - Пошаговое описание действий, приводящих к ошибке
- **Ожидаемое поведение** - Что вы ожидали увидеть
- **Фактическое поведение** - Что произошло на самом деле
- **Скриншоты** - Если возможно, приложите скриншоты, демонстрирующие проблему
- **Среда** - Информация о вашей операционной системе, версии Godot, версии проекта
- **Дополнительная информация** - Любая другая информация, которая может быть полезной

### Предложения по улучшению

#### Создание issue
Для предложений по улучшению создайте issue с:
- **Краткое описание** - Четкое описание предлагаемого улучшения
- **Мотивация** - Почему это улучшение будет полезным
- **Предлагаемая реализация** - Как можно реализовать это улучшение
- **Альтернативы** - Другие возможные подходы к решению проблемы

### Исправление ошибок

#### Выбор задачи
1. Просмотрите [issues с меткой "good first issue"](https://sourcecraft.dev/p2p-decentralized-emulation/forest-kingdoms-rpg/issues?q=is%3Aissue+is%3Aopen+label%3A%22good+first+issue%22) для начинающих
2. Выберите issue, над которым хотите поработать
3. Оставьте комментарий в issue, что вы начали работу над ней

#### Работа над исправлением
1. Форкните репозиторий
2. Создайте ветку для вашей работы:
   ```bash
   git checkout -b fix/issue-description
   ```
3. Внесите необходимые изменения
4. Добавьте тесты, если это необходимо
5. Обновите документацию, если это необходимо
6. Следуйте стандартам кодирования проекта

### Добавление новых функций

#### Обсуждение
1. Создайте issue с описанием предлагаемой функции
2. Обсудите предложение с командой разработчиков
3. Дождитесь одобрения перед началом реализации

#### Реализация
1. Создайте ветку для новой функции:
   ```bash
   git checkout -b feature/feature-name
   ```
2. Реализуйте функцию, следуя стандартам кодирования
3. Добавьте необходимые тесты
4. Обновите документацию
5. Создайте pull request

## Процесс разработки

### Рабочий процесс

#### 1. Форк и клонирование
```bash
# Форкните репозиторий через интерфейс GitHub
# Затем клонируйте свой форк
git clone https://sourcecraft.dev/your-username/forest-kingdoms-rpg.git
cd forest-kingdoms-rpg
```

#### 2. Настройка upstream
```bash
git remote add upstream https://sourcecraft.dev/p2p-decentralized-emulation/forest-kingdoms-rpg.git
```

#### 3. Создание ветки
```bash
# Для исправлений ошибок
git checkout -b fix/issue-description

# Для новых функций
git checkout -b feature/feature-name
```

#### 4. Работа над изменениями
1. Внесите необходимые изменения
2. Следуйте стандартам кодирования
3. Добавьте тесты
4. Обновите документацию

#### 5. Коммиты
Следуйте конвенции Conventional Commits:
```bash
# Примеры сообщений коммитов
git commit -m "fix: resolve player position sync issue"
git commit -m "feat: add new quest system"
git commit -m "docs: update gameplay mechanics documentation"
```

#### 6. Синхронизация с upstream
```bash
git fetch upstream
git rebase upstream/main
```

#### 7. Отправка изменений
```bash
git push origin your-branch-name
```

### Создание Pull Request

#### Перед созданием PR
1. Убедитесь, что ваш код проходит все тесты
2. Проверьте, что ваш код соответствует стандартам кодирования
3. Обновите документацию при необходимости
4. Убедитесь, что все изменения покрыты тестами

#### Создание PR
1. Перейдите в раздел Pull Requests оригинального репозитория
2. Нажмите "New Pull Request"
3. Выберите вашу ветку как source branch
4. Добавьте описание изменений:
   - Что было изменено
   - Почему это изменение необходимо
   - Как тестировать изменения
   - Скриншоты (если применимо)

#### Процесс ревью
1. Ожидайте комментариев от ревьюеров
2. Внесите необходимые изменения на основе комментариев
3. Ответьте на все комментарии
4. После одобрения PR будет смержен в основную ветку

## Стандарты кодирования

### GDScript
Следуйте стандартам, описанным в [coding-standards.md](coding-standards.md):
- Используйте snake_case для переменных и функций
- Используйте PascalCase для классов
- Добавляйте типы к переменным и параметрам
- Документируйте публичные методы

### JavaScript
Следуйте стандартам, описанным в [coding-standards.md](coding-standards.md):
- Используйте camelCase для переменных и функций
- Используйте PascalCase для классов
- Используйте async/await вместо промисов
- Добавляйте JSDoc комментарии к публичным методам

## Тестирование

### Запуск тестов

#### GDScript тесты
```bash
# Запуск всех тестов
godot --headless --run-tests

# Запуск конкретного теста
godot --headless --run-tests --test TestClassName
```

#### JavaScript тесты
```bash
# Запуск всех тестов
npm test

# Запуск тестов с покрытием
npm run test:coverage

# Запуск тестов в режиме watch
npm run test:watch
```

### Написание тестов

#### GDScript
```gdscript
# test_example.gd
extends GutTest

func test_example_function():
    var result = example_function(5, 3)
    assert_eq(result, 8)
```

#### JavaScript
```javascript
// example.test.js
import { exampleFunction } from './example.js';

describe('exampleFunction', () => {
    it('should add two numbers correctly', () => {
        expect(exampleFunction(5, 3)).toBe(8);
    });
});
```

## Стиль коммитов

Следуйте конвенции Conventional Commits:

### Типы коммитов
- **feat**: Новая функция
- **fix**: Исправление ошибки
- **docs**: Изменения в документации
- **style**: Изменения форматирования
- **refactor**: Рефакторинг кода
- **test**: Добавление или изменение тестов
- **chore**: Изменения в сборке или вспомогательных инструментах

### Примеры
```bash
git commit -m "feat: add new quest system"
git commit -m "fix: resolve player position sync issue"
git commit -m "docs: update gameplay mechanics documentation"
git commit -m "refactor: optimize collision detection algorithm"
```
